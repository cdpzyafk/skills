## 1. 第一原则

**套利行为**是锚定合约价格贴近现货的**实时原因**。资金费率是套利**不充分时**的补偿机制，是结果，不是原因。

```
正确因果链：
  基差出现 → 套利者检测 → 买卖对冲 → 基差收窄（实时，秒级）
  如果套利不足 → 溢价指数积累 → 资金费率信号（滞后，24h均值）→ 交易者调仓（慢）

错误因果链（常见误解）：
  资金费率高 → 所以合约价格下降
  （资金费率是结果，不是驱动力；真正的驱动力是套利者的实时买卖）
```

三层反馈的时间尺度：

| 层级 | 时间尺度 | 机制 | 触发条件 |
|------|---------|------|---------|
| **L1 实时层** | 每 tick（秒级） | 套利者检测基差，双腿对冲 | `|基差| > arb_threshold` |
| **L2 中频层** | 每5秒 | 溢价指数积累 → 资金费率信号 → 交易者调仓 | 溢价指数采样 |
| **L3 结算层** | 每8小时（典型） | 资金费率实际结算 → 仓位成本脉冲 → 持仓离场 | 结算时刻到达 |

## 2. 套利执行机制

### 正基差套利（合约溢价于现货）

```
触发条件：contract_bid - 指数价格 > arb_threshold_long

执行：
  合约侧：击打 bid（市价卖合约）→ bid 单向下移，ask 不受直接冲击 → 价差暂时扩大
  现货侧：买现货
  [闭环模式] 买现货推高指数价格 → 基差从两侧同步收窄

经济逻辑：
  合约>现货 → 卖合约+买现货 → 锁定无风险利差
  持仓期间需支付借贷成本（持有现货头寸的资金成本）
```

### 负基差套利（合约折价于现货）

```
触发条件：指数价格 - contract_ask > arb_threshold_short

执行：
  合约侧：击打 ask（市价买合约）→ ask 单向上移，bid 不受直接冲击 → 价差暂时扩大
  现货侧：卖现货（或做空现货）
  [闭环模式] 卖现货压低指数价格 → 基差从两侧同步收窄

经济逻辑：
  合约<现货 → 买合约+卖现货 → 锁定无风险利差
  卖出现货反而释放资金，无借贷成本（与正基差方向不对称）
```

**关键细节：** 每次套利只击打单侧（bid 或 ask），另一侧不受直接冲击，因此造成价差**暂时扩大**（不是缩小）。价差在下一周期由做市商重新报价后恢复。

## 3. 套利成本模型

### 正基差方向（持有现货需借贷）

```
arb_threshold_long = base_threshold_bps
                   + borrow_rate_bps_daily / 365 / 86400 × tick_interval_sec
                   + volatility_bps × volatility_multiplier
```

### 负基差方向（卖出现货释放资金）

```
arb_threshold_short = base_threshold_bps
                    + volatility_bps × volatility_multiplier
```

### 成本分量说明

| 成本分量 | 适用方向 | 含义 |
|---------|---------|------|
| `base_threshold_bps` | 双向 | 手续费 + 滑点的固定最低门槛 |
| `borrow_rate_bps_daily` | **仅正基差** | 持有现货头寸的借贷成本；负基差方向无此成本 |
| `volatility_bps` | 双向 | 近10秒指数价格波动率（实时更新） |
| `volatility_multiplier` | 双向 | 波动率对套利风险溢价的放大系数 |

### 不对称性的重要含义

- **正基差比负基差更难被套利抹平**——持有现货的借贷成本使正基差阈值天然更高
- 如果正负基差用同一个阈值，就低估了正基差的持续能力
- 设计文档或仿真代码中看到单一阈值，即为设计缺陷

## 4. 阈值动态变化

波动率变化直接影响套利是否介入：

```
波动率上升（如现货双向震荡）：
  volatility_bps ↑ → arb_threshold_long ↑ 且 arb_threshold_short ↑
  → 原本满足条件的基差（如 ±3bps）变为低于新阈值
  → 套利者全面退出 → L1 失效 → 基差开始自由漂移

波动率回落：
  volatility_bps ↓ → 阈值收缩
  → 积累的基差突然高于新阈值 → 套利者重新大批进入
  → "弹弓效应"：积累偏差被快速抹平，可能超调
```

**价差对阈值的间接消耗：** bid-ask spread 扩大后，实际可执行的套利利差缩小（需同时击打 bid 或 ask），等效于阈值升高，即使 `arb_threshold` 参数不变，也可能导致 L1 失效。

## 5. 闭环反馈

在闭环模式（`closed_loop`）下，套利行为不只影响合约盘口，也推动现货（指数）价格：

```
正基差套利 → 买现货 → 推高指数价格（index_feedback_ratio）
  → 合约侧 bid 下移 + 指数侧上移 → 基差从两侧同步收窄（比单向快）

负基差套利 → 卖现货 → 压低指数价格
  → 合约侧 ask 上移 + 指数侧下移 → 基差从两侧同步收窄
```

**index_feedback_ratio 的量级：** 现实中取值极小（约 0.02），原因：
- 现货市场深度是合约的 10–100 倍
- 指数由多个交易所聚合，单一交易所现货买卖只影响指数的一部分

这意味着：单次套利对合约盘口的冲击远大于对指数的冲击，基差主要从合约侧收窄。

## 6. 套利失效场景

### 场景 A：波动率飙升驱动退出（vol_arb_exit）

```
现货双向震荡（±2%，10秒周期）
  → volatility_bps 飙升 → 双向阈值均上升
  → 当前基差（±3bps）< 新阈值 → 套利者全退 → L1 失效
  → 合约盘口与指数解耦 → 基差自由漂移
  → 可能漂移至触发强平（即使无外部操纵）

平息后的弹弓效应：
  波动率回落 → 阈值收缩 → 套利者重入
  → 积累偏差被快速抹平（有超调风险）
```

### 场景 B：价差扩大消耗套利空间（mm_spread_spiral）

```
基差持续略高于阈值（套利未触发）
  → MM 感知方向性流量 → 扩价
  → bid-ask spread 扩大 → 等效阈值升高 → L1 更难触发
  → 正反馈：L1 失效 → 基差持续 → MM 继续扩价
  （L2 也被虚胖的溢价指数放大，进一步加速螺旋）
```

### 场景 C：虚假流动性欺骗套利者（liquidity_pile / mm_spoofing）

```
ask 侧堆满虚假挂单
  → 冲击买价被压低 → 溢价指数偏负（低估真实基差）
  → 套利者看到信号为"负基差"→ 反向开仓
  → 虚假挂单被撤 → 真实盘口暴露 → 套利者被迫亏损平仓

识别方法：
  比较 BBO mid 基差（标记价格用）vs 冲击价格基差（资金费率用）
  若两者方向相反（basis_signal_split = true），说明盘口存在失真
```

### 场景 D：套利者竞争（arb_crowding）

```
大基差出现 → 同时吸引大量套利者
  → 每人消耗基差量下降（竞争摊薄）
  → 边际套利者（成本稍高）进入后无法覆盖成本 → 亏损退出
  → 剩余套利者仍少于基差消耗所需 → 基差在半途停止收敛
  → 结果：收敛不完全，基差在阈值附近长期悬挂
```

### 场景 E：HFT 前跑（arb_race_front_run）

```
HFT 更快感知基差 → 抢先击打盘口
  → 普通套利者到达时基差已被部分消耗
  → 可得利差 < 普通套利者成本 → 无法进场
  → 有效阈值被隐性抬高（不是参数变了，是实际可执行的利差变小了）
  → 表面上基差仍高于名义阈值，但市场已无法收敛
```

### 场景 F：资本耗尽（arb_exhaustion）

```
持续正基差 → 套利者长期持有现货对冲仓
  → 借贷成本持续累积 → 净收益逐渐缩小
  → 即使基差未消失，资本耗尽后套利者主动离场
  → 基差在 arb_threshold 以上长期悬挂
  → 这是资本约束问题，不是阈值计算问题
```

## 7. 套利生命周期

```
阶段 1：机会发现
  基差 > arb_threshold → 套利者检测到 → 准备进场

阶段 2：积极套利（正常收敛）
  套利者执行双腿 → 盘口收敛 → 基差缩小

阶段 3：竞争与摊薄（可能）
  多个套利者同时进场 → 边际套利者被挤出 → 收敛速度下降

阶段 4：空间消失
  基差 < arb_threshold → 套利者全退 → 基差在阈值附近稳定

阶段 5：失效或反转（异常情况）
  波动率上升 → 阈值抬高 → 重新打开空间
  或盘口操纵 → 虚假信号 → 套利者被诱骗进错误方向
  或资本耗尽 → 套利者被迫离场 → 基差持续
```

## 8. 现实参数范围（BTC 主流合约）

| 参数 | 现实值 | 压力测试值 | 说明 |
|------|--------|-----------|------|
| `base_threshold_bps` | 3–5 bps | 50+ bps | 手续费+滑点；低效市场更高 |
| `arb_strength` | 0.05–0.15/tick | 0.8–1.0 | 每秒关闭基差的比例；**0.8 严重高估**，仅 HFT 接近 1 |
| `borrow_rate_bps_daily` | 5–15 bps/day | 50+ bps/day | 正基差方向借贷成本；牛市融资成本更高 |
| `volatility_multiplier` | 1.5–3.0 | 5.0+ | 波动率放大系数；极端行情下套利者风险溢价更高 |
| `index_feedback_ratio` | 0.01–0.05 | 0.1+ | 套利对指数的反推比例；现货深度越深比例越小 |
| `depth_usdt` | $5M–$50M | $100K | BTC 主流合约盘口深度 |
| `impact_amount_usdt` | $200（Binance BTC） | $1000+ | 冲击价格计算用名义金额 |

**最常见的参数错误：**
- `arb_strength = 0.8`：意味着每秒关闭80%的基差，远超现实（实测5-15%）
- `impact_amount_usdt = 1000` 配合 `depth_usdt = 5000000`：冲击 0.02%，溢价指数接近零，资金费率信号无法产生

## 快速核查：评审合约设计文档时的套利相关检查项

```
□ 套利是否被描述为"实时"机制？（不是5秒均值，不是资金费率驱动）
□ 正负基差阈值是否分开计算？（正基差需含借贷成本，负基差不需要）
□ volatility_bps 是否动态更新并影响阈值？（静态阈值无法反映高波动率下的套利退出）
□ arb_strength 是否在 0.05–0.15 范围内？（0.8 = HFT，需要有充分理由）
□ impact_amount_usdt 是否与 depth_usdt 匹配？（比值决定冲击幅度，不能独立设置）
□ 闭环模式的 index_feedback_ratio 是否足够小？（现货深度是合约的10-100倍）
□ 是否有"套利者退出"场景的建模？（vol_arb_exit 类型的场景）
□ 盘口失真（liquidity_pile/thin_book）是否导致套利者误判？
□ 是否混淆了"名义阈值未触发"和"实际可执行利差不足"？（HFT前跑导致后者）
```
